# DB Migration Drift — Audit & Remediation Plan (tracked, deferred)

- **Status:** Diagnosis **complete** (Session #11, 2026-07-03, evidence-backed). Remediation **deferred** to a dedicated future session. **Nothing here is fixed yet.**
- **TL;DR:** Alembic was effectively **abandoned** early. Dev builds the DB entirely via `Base.metadata.create_all`; the migration chain never kept up. `recommendations.feedback_at` (the column that first surfaced this) was only the *symptom* — there are four compounding problems below.
- **⚠️ Do NOT half-do the remediation.** A partial fix (e.g. merging heads but not catching up the 32 missing tables, or stamping a mismatched DB) can leave the migration state **worse** than the honestly-abandoned state it's in now. Remediate all-or-nothing, deliberately.

## The four problems (evidence)

1. **Branched chain — two heads, no merge.** `001 → af_20260504_0001 →` splits into **both** `df_20260505_0001` **and** `rec_20260505_0001` (each `down_revision = af_20260504_0001`). `alembic heads` → `['df_20260505_0001', 'rec_20260505_0001']`. `alembic upgrade head` (singular) therefore errors: *"Multiple head revisions are present."*
2. **No `alembic.ini`** anywhere in the repo — the alembic CLI is not wired to run (only invocable via the Python API). `backend/alembic/env.py` exists and is async-aware (`async_engine_from_config`, reads `settings.database_url`).
3. **Migrations cover 18 of 50 model tables — 32 have NO migration.** Missing (need autogenerated migrations): `api_usage, certifications, compliance_audit_logs, defensive_gameplans, defensive_priorities, eafc_formations, eafc_squads, fn_build_sessions, fn_edit_drills, fn_player_twins, fn_zone_rotations, identity_profiles, mlb_lineup_slots, mlb_lineups, mlb_players, notifications, player_twins, push_subscriptions, referrals, responsible_gambling_configs, secret_weapons, support_tickets, ufc_fighters, ufc_rounds, undisputed_fighters, undisputed_fights, user_arsenal, vault_entries, video_poker_sessions, warzone_loadouts, warzone_matches, weapon_usage_log`. (0 orphaned migration tables — nothing migrated-but-dropped-from-models.)
4. **The dev `esportsforge.db` was never stamped by alembic** — `alembic current` on a copy is empty. It was built purely by `create_all`, so it carries no `alembic_version`.

## Throwaway upgrade test (Session #11)

On a **disposable copy** of `esportsforge.db` (real DB never touched), via the alembic Python API:
- `alembic upgrade head` → **FAILED: `CommandError: Multiple head revisions are present`** (problem 1).
- Even with the branch resolved, upgrading a never-stamped, `create_all`-built DB **from base** would collide — `001_initial_schema` would recreate tables that already exist (problems 3 + 4).

**Conclusion: `alembic upgrade head` is not a viable fix as-is.** "Just run the migration" was disproven on the copy.

## Remediation outline (the future scoped project — all-or-nothing)

1. Add `alembic.ini` (point `script_location` at `backend/alembic`).
2. **Merge the two heads** (`alembic merge -m … df_20260505_0001 rec_20260505_0001`) or otherwise linearize.
3. **Autogenerate migrations** to catch up the 32 un-migrated tables so the chain reproduces the current models.
4. For existing `create_all`-built DBs whose schema already matches head: `alembic stamp head` (do **not** run `upgrade` from base — it collides). Repair genuine column drifts (e.g. `feedback_at`) explicitly.
5. Switch the app lifespan from `create_all` to `alembic upgrade head` (optionally keep `create_all` only for ephemeral test DBs).

**Interim (until remediation):** `create_all` works for dev/test — keep using it. **Never run `alembic upgrade` against a `create_all` DB** (it fails/collides). Use a throwaway `DATABASE_URL` for any migration experiments.
