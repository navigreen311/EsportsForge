# Capture Agent — Specification

> Companion: [00-overview.md](00-overview.md), [02-core-service-spec.md](02-core-service-spec.md).
> **The capture agent is universal. It knows nothing about which game the player is running.** Title detection is server-side.

## Purpose

The capture agent is a small native binary that runs on the player's PC. It pulls frames from a configured video source (HDMI capture card, screen region, or test video file), packages them, and ships them to the VisionAudioForge core service over a persistent connection.

Players run it once and forget it. It launches with EsportsForge, reads its config, validates the capture source, and stays alive for the session.

## Scope

In scope:
- Frame capture from HDMI capture cards (Elgato, AVerMedia, generic UVC) on Windows
- Screen capture from a configured region (for PC players)
- Frame compression and adaptive sampling
- Authenticated streaming to the core service
- Local ring buffer for momentary disconnects
- Health/heartbeat reporting
- Hot-reload of integrity-mode signals from the core (capture pause/resume)
- Logging and minimal in-app diagnostics UI

Out of scope (v1):
- Mac / Linux support (Windows-only for the CLX PC use case)
- Image processing of any kind (no OCR, no detection — that's the core's job)
- Title detection (core's job)
- Audio capture (deferred to v2 — VisionAudioForge name is forward-looking)
- Multi-monitor / dual capture (single source per session in v1)

## Recommendation: stack and distribution

**Language: Python 3.12 + OpenCV + websockets, packaged with PyInstaller.**

Why Python:
- OpenCV bindings are first-class. Capture-card access via `cv2.VideoCapture(index, cv2.CAP_DSHOW)` is one line on Windows.
- PyInstaller produces a single `.exe` the player downloads. No system Python required.
- The team already maintains Python on the backend; one less language to support.
- Frame-level perf is fine — JPEG-encoding a 1080p frame at 10–15 fps is negligible CPU.

Why not Electron:
- Heavier install (~150 MB vs ~30 MB), slower startup.
- Capture-card DirectShow access from Node requires a native module anyway.
- We don't need a UI framework — diagnostic UI is one window with status lines.

Why not native C++ / Rust:
- Faster, but ~3x dev cost. Capture is not the bottleneck — network + server is.
- We'll port to Rust later if frame rate becomes the limit.

Distribution:
- Auto-updating Windows installer (NSIS or Inno Setup).
- Signed with EV cert (anti-cheat platforms get nervous about unsigned binaries that read framebuffers).
- Default install path `%LOCALAPPDATA%\EsportsForge\CaptureAgent\`.

## Capture sources

The agent supports three sources, selectable via config or the in-app "Capture Source" picker (the existing `CaptureSourceModal` in the frontend already presents these three):

### 1. `capture-card` — HDMI capture card

Player connects PS5/Xbox HDMI out → capture card HDMI in → capture card USB to CLX PC. Capture card appears as a UVC device.

Implementation: `cv2.VideoCapture(device_index, cv2.CAP_DSHOW)`.

Detection: enumerate UVC devices on startup. If exactly one HDMI-capable device is present, auto-select. Otherwise, prompt the player.

Tested cards (must be in the v1 release notes):
- Elgato HD60 X (USB 3.0)
- Elgato Game Capture 4K X
- AVerMedia Live Gamer 4K
- Generic MS2109-based USB UVC capture (e.g., "USB3 HDMI Capture")

### 2. `pc-monitor` — screen region capture

Player runs the game natively on the same PC. Agent grabs a configured monitor region.

Implementation: `mss` library (fast, Windows-friendly), configurable monitor index + crop rect. Default = full primary monitor.

Anti-cheat consideration: many ranked PC games (Valorant, Warzone) detect screen-grab tools via Vanguard / Ricochet kernel hooks. **The agent must not run on the same PC during ranked play in those titles.** This is enforced in the core service's integrity gate (Doc #02), but the agent surfaces a warning when starting a session for a flagged title with `pc-monitor` source selected.

### 3. `test-video` — file-based, for development

Plays back a pre-recorded MP4 to the capture pipeline. Used for repro testing, CI, and adapter development without needing the actual console.

Config-only — never exposed in production UI.

## Configuration

A single TOML file at `%LOCALAPPDATA%\EsportsForge\CaptureAgent\config.toml`:

```toml
# Generated by the EsportsForge desktop installer / Settings → Game Settings.
[auth]
api_key = "esf-cap-..."          # bound to user, rotates every 90 days
user_id = "5041bbe7-..."

[core]
endpoint = "wss://vision.esportsforge.gg/ws/ingest"
fallback_endpoint = "ws://localhost:8100/ws/ingest"  # for dev

[capture]
source = "capture-card"          # capture-card | pc-monitor | test-video
device_index = 0                 # for capture-card
monitor_index = 0                # for pc-monitor
crop = [0, 0, 1920, 1080]        # for pc-monitor
test_video_path = ""             # for test-video

[transport]
target_fps = 12                  # base frame rate sent to core
adaptive = true                  # bump FPS on motion
adaptive_max_fps = 24
jpeg_quality = 75                # 0-100; 75 is a good balance
batch_size = 4                   # frames per WS message
ring_buffer_seconds = 30         # local replay if core blips

[diagnostics]
log_level = "INFO"
log_path = "%LOCALAPPDATA%/EsportsForge/CaptureAgent/logs/agent.log"
```

The desktop installer writes this file from the player's Game Settings selections. The agent reads it on startup; SIGHUP / file-watcher hot-reload is **not** in v1 (restart agent to apply changes).

## Wire protocol

### Connection lifecycle

WebSocket to `wss://vision.esportsforge.gg/ws/ingest`. Auth via `Authorization: Bearer <api_key>` header on the upgrade request.

Connect handshake (server sends first message):

```json
{
  "type": "session_open",
  "session_id": "ses_01HXXX",
  "integrity_mode": "ranked",        // current mode from EsportsForge
  "active_title": "madden26",        // hint from EsportsForge active-title setting
  "capture_allowed": true,
  "frame_format": "jpeg",
  "max_fps": 24
}
```

If `capture_allowed` is `false` (e.g., Tournament mode), the agent stays connected, sends heartbeats, but does not send frames until the core publishes a `capture_resume` event.

### Frame batches (agent → core)

```json
{
  "type": "frame_batch",
  "session_id": "ses_01HXXX",
  "batch_seq": 42,
  "frames": [
    {
      "frame_id": 1247,
      "captured_at": "2026-05-06T22:30:01.123Z",
      "width": 1920,
      "height": 1080,
      "format": "jpeg",
      "data_b64": "/9j/4AAQ..."     // base64 JPEG payload
    },
    ...
  ]
}
```

Payload size budget per WS message: **2 MB**. JPEG at q=75, 1080p ≈ 150 KB. Batch of 4 ≈ 600 KB. Plenty of headroom.

### Heartbeat (agent → core, every 5s)

```json
{
  "type": "heartbeat",
  "session_id": "ses_01HXXX",
  "stats": {
    "frames_captured": 4382,
    "frames_sent": 4380,
    "frames_dropped": 2,
    "current_fps": 12.1,
    "capture_device_status": "ok",
    "uptime_sec": 364
  }
}
```

### Control messages (core → agent)

```json
{ "type": "capture_pause", "reason": "integrity_mode_switch" }
{ "type": "capture_resume" }
{ "type": "set_target_fps", "fps": 18 }
{ "type": "session_close", "reason": "user_logout" }
```

## Adaptive sampling

The agent doesn't blindly send every frame at the configured `target_fps`. It uses a cheap motion gate:

1. Capture at 30 fps internally (for buffer freshness).
2. Default cadence: every Nth frame (where N = 30 / target_fps) gets queued for sending.
3. Motion gate (optional, `adaptive = true`): compute frame-difference against the previous queued frame. If diff is below threshold (static menu, paused game), drop down to 2 fps. If diff spikes above threshold (snap, kill, animation), bump up to `adaptive_max_fps` for the next 2 seconds.

This keeps bandwidth low during menus and ramps up during gameplay. Important for tournament venues with sketchy wifi.

## Local ring buffer

Last 30 seconds of frames held in memory (in compressed form). On WS disconnect, the agent does not drop new captures — it keeps queueing into the ring buffer. On reconnect, it sends the buffer contents in order before resuming live capture.

Ring buffer size cap: 30s × 24 fps × 200 KB ≈ 144 MB max. Acceptable on a gaming PC.

If the buffer fills (>30s disconnect), oldest frames are dropped. The agent reports `frames_dropped` in heartbeat once reconnected.

## Diagnostic UI

Single status window (Tk-based or PyQt). Shows:
- Connection status (connected / reconnecting / paused by integrity mode)
- Active title (from server hint)
- Capture FPS / sent FPS / dropped count
- Capture device name
- Test capture button (saves a single frame to disk for debugging)
- "Open log file" button

Window can be minimized to system tray. Closing the window closes the agent.

## Observability

Local log file rotates at 10 MB, keeps 5 generations. INFO level by default; DEBUG via env var `ESF_CAPTURE_LOG_LEVEL=DEBUG`.

The agent does not phone home metrics directly. The core service tracks per-session metrics from the heartbeat stream.

## Security and trust model

The agent runs on the player's machine — it is **not trusted** by the core. Concretely:

- **API keys are short-lived (90 days) and bound to a user_id.** Rotation handled in EsportsForge backend; agent fetches fresh key on next launch.
- **The core re-validates every frame's session.** A leaked API key cannot stream frames for a different user.
- **Anti-cheat enforcement is server-side.** A modified agent cannot bypass Tournament-mode capture restrictions because the core gates processing on the user's current Integrity Mode (queried fresh from EsportsForge backend, not trusted from the agent).
- **No frames are persisted in v1.** The agent is stateless across launches (logs only).

## Failure modes and what the agent does

| Failure | Behavior |
|---|---|
| Capture device disconnected | Heartbeat reports `capture_device_status: "missing"`, frame send pauses, diagnostic UI shows red. Auto-recovers when device returns. |
| WS disconnect | Reconnect with exponential backoff (1s → 2s → 4s → max 30s). Frames queue in ring buffer. |
| Auth failure (401) | Stop trying to reconnect. Diagnostic UI prompts player to re-launch from EsportsForge desktop. |
| Frame encode failure | Drop the single frame, increment dropped counter, continue. |
| Config file missing | Refuse to start. Show "EsportsForge needs to set up Capture Agent — open Settings → Game Settings → Capture Source." |

## What this spec does not decide

- **Bundling with the Next.js app.** Whether the desktop installer that ships the capture agent is a separate download or wrapped into a future EsportsForge desktop app is a product/distribution decision out of scope here. The agent is built to run standalone either way.
- **Audio capture.** "VisionAudioForge" is forward-looking — v1 is video only. Audio (game commentary OCR, callout detection) is a v2 stretch and gets its own spec when scoped.
- **Mobile capture.** Out of scope. Mobile titles aren't in EsportsForge's roster.
